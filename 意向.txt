## 个人 AI 工作台 — 需求详细分析

### 一、市场格局确认（2026 年中）

| 层级 | 已有玩家 | 状态 | 缝隙 |
|------|---------|------|------|
| 本地推理引擎 | Ollama、LM Studio、llama.cpp | 已定局 | 无 |
| API 网关/路由 | LiteLLM（16k Star）、One API（20k Star） | 已定局 | 无 |
| 本地聊天 | GPT4All（百万级安装）、Ollama WebUI | 已定局 | 无 |
| 本地文档问答 | Kotaemon（25k Star）、Anything-LLM | **有玩家但未定局** | **有** |
| 个人知识管理+AI | 无成熟产品 | **空白** | **有** |

关键判断：纯聊天已被 Ollama + GPT4All 覆盖，但"聊天 + 文档问答 + 知识管理"的一体化个人工作台，尚无一个产品做到"开箱即用、体验流畅"。

---

### 二、目标用户画像

**核心用户**：有大量本地文档、重视隐私、不想折腾技术栈的个人知识工作者。

| 用户类型 | 典型场景 | 核心痛点 |
|---------|---------|---------|
| 科研人员/学生 | 管理 50+ 篇论文 PDF，快速定位某篇的某个结论 | PDF 解析差、检索不准、无法溯源 |
| 律师/法务 | 合同条款检索、判例对比 | 隐私要求极高，不能上传云端 |
| 产品经理/技术写作者 | 需求文档、技术规范、会议纪要堆积 | 跨文档关联查询困难 |
| 自由职业者/创作者 | 笔记、灵感、素材散落各处 | 缺乏统一的知识入口 |
| 普通知识工作者 | 日常文档、邮件、笔记 | 不想学 Docker、不想配 RAG 参数 |

---

### 三、竞品深度对比

| 维度 | GPT4All | Kotaemon | Anything-LLM | Ollama + Open WebUI | **我们的机会** |
|------|---------|----------|-------------|---------------------|---------------|
| 安装难度 | 一键安装 | Docker | Docker/桌面版 | 两步（Ollama + WebUI） | **一键安装** |
| 本地聊天 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 文档问答 | LocalDocs（弱） | ✅ 混合检索 | ✅ | ✅（需配置） | **核心差异化** |
| PDF 解析质量 | 一般 | 较好 | 较好 | 依赖配置 | **重点突破** |
| 引用溯源 | 无 | ✅ 页码级 | ✅ | 部分 | ✅ |
| 知识库管理 | 无 | 文件集合 | 工作区 | 无 | **核心差异化** |
| 多格式支持 | PDF/TXT | PDF/Word/TXT/图片 | 多种 | 依赖配置 | PDF/Word/PPT/Excel/图片 |
| 本地模型 | ✅ | ✅（Ollama） | ✅ | ✅ | ✅（已有 llama.cpp） |
| 开源协议 | MIT | Apache 2.0 | MIT | MIT | 待定 |
| 社区/生态 | 大 | 增长中 | 中等 | 大 | 从零 |

**结论**：竞品在"文档问答"上都有产品，但体验割裂——要么安装复杂（Docker），要么 PDF 解析质量差，要么缺乏知识库管理。**一体化 + 开箱即用 + 优质 PDF 解析**是突破口。

---

### 四、用户核心痛点分层

#### P0 — 不解决就没法用

| 痛点 | 描述 | 竞品现状 |
|------|------|---------|
| PDF 解析质量差 | 多栏排版、表格、公式解析后变成乱码，RAG 检索准确率极低 | 所有竞品都有此问题 |
| 安装配置门槛高 | 需要装 Docker、配向量数据库、调 RAG 参数 | Kotaemon/Anything-LLM 需要 Docker |
| 检索不精准 | 纯向量检索漏关键词，纯关键词检索漏语义 | 多数竞品只用向量检索 |
| 无法溯源 | AI 回答看起来合理但不知道出自哪里，无法验证 | GPT4All 无此功能 |

#### P1 — 严重影响体验

| 痛点 | 描述 |
|------|------|
| 文档散落无组织 | 文件在文件夹里，和 AI 对话是两套系统，无法统一管理 |
| 跨文档无法关联 | 问"合同 A 和合同 B 的违约责任条款有什么区别"——竞品做不到 |
| 不支持表格/图表理解 | 财务报表、实验数据表格无法被正确理解和回答 |
| 模型切换不直观 | 想换模型要改配置、重启，非技术用户不会操作 |

#### P2 — 锦上添花

| 痛点 | 描述 |
|------|------|
| 无知识库分类/标签 | 文档多了以后无法按主题、项目、时间组织 |
| 无对话历史管理 | 聊天记录无法保存、搜索、复用 |
| 无导出/分享 | 问答结果无法导出为笔记或报告 |
| 无定时/自动处理 | 不能自动监控文件夹、自动索引新文档 |

### 五、功能需求分层

#### MVP（最小可行产品）— 2~3 周

```
本地聊天 + 拖入文档提问
```

| 功能 | 详细说明 | 优先级理由 |
|------|---------|-----------|
| 本地模型推理 | 基于现有 llama.cpp 封装，支持 GGUF 模型加载和热切换 | 已有基础 |
| 聊天界面 | 基于现有 WebUI 改造，支持流式输出、Markdown 渲染 | 已有基础 |
| 文档上传解析 | 支持 PDF/Word/TXT，拖拽上传，自动分段 | **核心新增** |
| 文档问答 | 基于上传文档的 RAG 问答，带引用溯源 | **核心新增** |
| 一键安装包 | Windows/macOS 单文件安装，内置一个小模型 | **关键差异化** |

#### V1.0 — 4~6 周

```
知识库管理 + 混合检索 + 多格式
```

| 功能 | 详细说明 |
|------|---------|
| 知识库管理 | 创建/删除/重命名知识库，文档归类，标签系统 |
| 混合检索 | 全文检索 + 向量检索 + 重排序（BM25 + Embedding + Cross-encoder） |
| 多格式支持 | PPT/Excel/图片（OCR）/Markdown |
| 表格理解 | 对文档内表格的结构化提取和问答 |
| 跨文档问答 | 在同一个知识库内跨多份文档检索和回答 |
| 模型管理 | 内置模型下载、切换、删除，可视化配置 |

#### V2.0 — 8~12 周

```
智能知识管理 + Agent 自动化
```

| 功能 | 详细说明 |
|------|---------|
| 智能分类 | 上传文档自动识别类型、提取关键词、建议标签 |
| 知识图谱 | 文档间关联关系可视化，概念提取 |
| 对话历史 | 保存/搜索/复用历史对话 |
| 导出报告 | 基于知识库内容生成摘要、对比报告 |
| 文件夹监控 | 自动索引指定文件夹的新增文档 |
| Agent 自动化 | 定时总结、自动归类、批量问答 |

### 六、技术方案关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| PDF 解析 | 自研解析管线（PyMuPDF + 布局识别 + 表格提取） | 这是核心壁垒，不能依赖第三方 |
| 向量数据库 | LanceDB（嵌入式）或 ChromaDB | 零配置、嵌入式，符合"一键安装"理念 |
| Embedding 模型 | 内置轻量模型（如 bge-small-zh）+ 支持自定义 | 开箱即用 + 可升级 |
| RAG 策略 | 混合检索（BM25 + 向量）+ Cross-encoder 重排 | 竞品验证过的有效方案 |
| 前端框架 | 保留现有 WebUI 基础，渐进改造 | 复用 70% 代码 |
| 安装方式 | PyInstaller 打包为单文件 exe/dmg | 对标 GPT4All 体验 |
| 内置模型 | 打包一个 1-2B 的轻量模型 | 安装后立即可用 |

### 七、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| PDF 解析是深坑 | 核心功能不可用 | MVP 阶段先支持排版简单的 PDF，复杂排版逐步迭代 |
| 内置模型效果差 | 用户第一印象差 | 提供一键下载更大模型的入口，引导用户升级 |
| 竞品快速跟进 | 差异化窗口缩短 | 聚焦"知识库管理"这个竞品薄弱的环节建立壁垒 |
| 安装包体积大 | 下载门槛高 | 提供"轻量版（不含模型）"和"完整版"两个选项 |
| 跨平台兼容 | 维护成本高 | MVP 先做 Windows，macOS 紧随其后 |

### 八、一句话定位

> **"把你自己电脑上的所有文档，变成一个可以对话的私人图书馆——不需要上传任何东西到云端，不需要学 Docker，不需要配参数。"**
